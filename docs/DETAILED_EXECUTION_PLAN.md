# 제품 관리 앱 상세 실행계획

이 문서는 `PRODUCT_VISION_AND_ROADMAP.md`의 방향을 실제 개발 작업으로 옮기기 위한 실행계획이다.

방향성 문서가 "무엇을 만들 것인가"를 정한다면, 이 문서는 "어떤 순서로 무엇을 수정하고, 어느 상태가 되면 완료로 볼 것인가"를 정한다.

---

## 1. 현재 위치

### 이미 구현된 기반

- Flutter Material 3 앱
- 홈, 내 제품, 기록, 자랑 하단 탭
- 공간별 제품 목록
- 공간 추가와 삭제
- 일반 제품 등록
- 브랜드와 모델명 등록
- 제품명, 브랜드, 모델명 검색
- 서버 검색 실패 시 앱 내장 카탈로그 폴백
- 제품별 기본 관리법
- 관리법 출처 유형 표시
- 출처 URL과 확인일 저장
- 제품 스펙 표시
- 공식 또는 참고 영상 연결
- 특정 청소용품 추천
- 관리 완료 기록
- FastAPI 제품 카탈로그 API
- `draft`, `reviewed`, `verified` 검수 상태
- 검수된 JSON 제품 데이터
- Flutter 및 서버 자동 테스트의 기본 골격

### 현재 구현의 한계

- 서버 카탈로그의 제품 수가 매우 적다.
- 앱 내장 카탈로그와 서버 카탈로그가 중복 관리되고 있다.
- 같은 제품을 여러 번 등록할 수 있다.
- 검색 결과의 정렬 기준과 일치 수준이 충분히 명확하지 않다.
- 제품 등록 과정이 한 화면에 길게 이어진다.
- 제품 종류와 정확한 모델 등록의 관계가 복잡하게 느껴질 수 있다.
- 검색 결과가 없을 때 정보 요청으로 연결되지 않는다.
- 제품 상세 화면이 긴 단일 목록 구조다.
- 관리법이 평상시 관리, 문제 해결, 소모품으로 나뉘어 있지 않다.
- 관리 기록은 제품별 상세 이력으로 보기 어렵다.
- 소모품과 필터 교체 기록이 별도 개념으로 존재하지 않는다.
- 사용자 메모, 사진, 구매일, 설치일, 보증 정보가 없다.
- 서버 데이터는 JSON 파일이며 운영자 수정 화면이 없다.
- 계정과 클라우드 동기화가 없다.
- 검색 실패, 인기 제품, 미등록 모델에 대한 운영 데이터가 없다.
- 자랑 탭은 제품 관리 방향과 아직 충분히 연결되지 않는다.

---

## 2. 개발 원칙

### 핵심 경험을 먼저 완성한다

모든 개발 우선순위는 다음 사용자 흐름을 기준으로 정한다.

1. 사용자가 자기 제품을 찾는다.
2. 제품을 어렵지 않게 등록한다.
3. 신뢰 수준이 표시된 관리 정보를 확인한다.
4. 실제로 제품을 관리한다.
5. 관리 결과를 기록한다.
6. 나중에 다시 필요한 정보를 찾는다.

이 흐름에 직접 기여하지 않는 기능은 후순위로 둔다.

### 정확하지 않은 정보를 정확한 것처럼 보이지 않는다

- 동일 모델 공식 정보
- 동일 모델 비공식 공개 정보
- 같은 브랜드의 유사 모델
- 같은 처리방식의 유사 제품군
- 일반 제품군 관리법
- 사용자 직접 입력

정보가 어느 단계에 해당하는지 UI와 데이터에 함께 기록한다.

### 사용자가 막히지 않게 한다

제품을 찾지 못해도 등록할 수 있어야 한다.

모델명을 몰라도 제품 종류만 등록할 수 있어야 한다.

다만 일반 제품 등록과 정확한 모델 등록의 차이는 사용자에게 분명하게 설명한다.

### 앱을 매일 사용하도록 강요하지 않는다

알림, 일정, 완료율은 보조 기능이다.

제품 정보 탐색과 관리 기록이 앱의 중심이다.

# 3. 개발 단계 개요

## Phase 0. 기존 구조 정리와 기준선 고정

목표:

> 앞으로 제품 관리 기능을 추가할 때 예전 청소 체크 구조가 발목을 잡지 않도록 코드와 용어를 정리한다.

## Phase 1. 제품 검색과 등록 경험 완성

목표:

> 사용자가 모델명을 알든 모르든 막힘 없이 제품을 등록한다.

## Phase 2. 제품 카탈로그 품질과 운영 기반 강화

목표:

> 제품 정보가 많아져도 출처, 검수 상태, 버전을 안정적으로 관리한다.

## Phase 3. 제품 상세 화면 재설계

목표:

> 사용자가 필요한 관리 정보를 몇 초 안에 찾는다.

## Phase 4. 제품별 관리 기록 완성

목표:

> 단순 완료 기록을 제품 관리 이력으로 발전시킨다.

## Phase 5. 소모품과 필터 관리 (핵심 완료)

목표:

> 제품 관리에서 반복적으로 필요한 소모품 교체를 별도 기능으로 관리한다.

## Phase 6. 사용자 제보와 운영자 검수 (핵심 완료)

목표:

> 카탈로그에 없는 제품을 사용자와 운영자가 함께 확장한다.

## Phase 7. 데이터베이스와 관리자 도구

목표:

> JSON 파일을 벗어나 실제 서비스 운영이 가능한 데이터 구조를 만든다.

현재는 DB 전환 기준에 도달하지 않아 보류한다. 그 전까지 사용할 제보 상태 변경,
감사 이력과 카탈로그 품질 보고서 CLI는 구현했다. 상세 상태는
`PHASE_6_USER_SUBMISSIONS.md`와 `EXTERNAL_WORK_AND_USER_ACTIONS.md`에서 관리한다.

## Phase 8. 계정, 동기화와 가족 공유

목표:

> 제품과 기록을 여러 기기 및 가족 구성원과 안전하게 공유한다.

## Phase 9. 선택적 알림

목표:

> 사용자가 원하는 관리 항목만 부담 없이 알려준다.

## Phase 10. 추천용품과 수익화

목표:

> 신뢰를 해치지 않으면서 실제 관리에 필요한 제품을 연결한다.

## Phase 11. 제품별 커뮤니티와 문제 진단

목표:

> 충분한 데이터가 쌓인 뒤 제품별 경험과 문제 해결을 확장한다.

## Phase 12. 출시 준비

목표:

> 기능 추가를 멈추고 안정성, 정책, 실제 기기 품질을 확보한다.

# 4. Phase 0: 기존 구조 정리 (완료)

완료일: 2026-06-09

구현 기준과 호환성 결정은 `PHASE_0_PRODUCT_BASELINE.md`에 기록했다.

## 4.1 용어 통일

수정 대상:

- `구역`과 `공간`이 혼용되는 표현
- `항목`, `제품`, `가전 또는 가구`가 혼용되는 표현
- `청소 완료`, `관리 완료`, `기록 완료`가 혼용되는 표현

권장 기준:

- 사용자가 제품을 분류하는 장소: `공간`
- 사용자가 등록하는 대상: `제품`
- 모델 정보가 없는 대상도 화면에서는 `제품`으로 표현
- 실제 행동 결과: `관리 기록`
- 청소, 필터 교체, 점검을 포함하는 상위 개념: `관리`

완료 조건:

- 주요 화면에서 같은 개념에 같은 단어를 사용한다.
- 코드 내부 이름을 한 번에 모두 바꾸지는 않더라도 사용자 표시 문구는 통일한다.

## 4.2 예전 청소 체크 기능 정리

검토 대상:

- `today_screen.dart`
- `cleaning_session_screen.dart`
- `cleaning_task.dart`
- `cleaning_task_repository.dart`
- 체크리스트 전용 위젯
- 사용되지 않는 검색 위젯

작업:

- 현재 앱에서 실제로 참조되는지 확인한다.
- 제품 관리 방향에서 재사용할 기능과 제거할 기능을 분리한다.
- 사용하지 않는 파일은 테스트 확인 후 삭제한다.
- 관리 기록에 재사용 가능한 로직은 제품 관리 서비스로 이동한다.

완료 조건:

- 앱 진입 경로에 예전 `오늘 청소` 로직이 남아 있지 않다.
- 사용되지 않는 파일과 위젯이 제거된다.
- Flutter 분석과 테스트가 통과한다.

## 4.3 데이터 모델 역할 분리

현재 `ZoneItem`은 다음 책임을 동시에 가진다.

- 사용자 등록 제품
- 카탈로그 제품 정보
- 관리법
- 출처
- 일정
- 추천용품

장기적으로 분리할 모델:

- `CatalogProduct`: 서버가 관리하는 제품 정보
- `UserProduct`: 사용자가 자기 집에 등록한 제품
- `CareGuide`: 관리법과 주의사항
- `CareRecord`: 사용자의 관리 기록
- `Consumable`: 필터와 소모품
- `SourceReference`: 출처와 확인일

초기 작업:

- 바로 대규모 마이그레이션하지 않는다.
- 새 기능을 추가할 때 분리 모델을 기준으로 설계한다.
- 기존 저장 데이터 호환을 위한 마이그레이션 버전을 둔다.

완료 조건:

- 서버 모델과 사용자 제품 모델의 역할이 문서와 코드에서 구분된다.
- 사용자 제품이 카탈로그 제품 ID를 참조할 수 있다.

# 5. Phase 1: 제품 검색과 등록 경험 완성 (완료)

구현 상태는 `PHASE_1_REGISTRATION.md`에 기록한다.
실기기와 운영 환경이 필요한 후속 항목은
`PHASE_1_5_DEFERRED_VALIDATION.md`로 분리한다.

## 5.1 등록 흐름을 단계형으로 변경

현재 문제:

- 제품 검색, 종류, 주기, 시간, 브랜드, 모델이 한 화면에 길게 배치된다.
- 사용자가 무엇부터 해야 하는지 알기 어렵다.

권장 흐름:

### Step 1. 제품 찾기

- 검색창
- 최근 검색
- 인기 제품군
- 제품 종류 빠른 선택
- `모델명을 몰라요` 선택

### Step 2. 제품 확인

- 제품명
- 브랜드
- 모델명
- 주요 스펙
- 일치 수준
- 출처 상태
- `이 제품이 맞아요`

### Step 3. 내 제품 정보

- 공간 선택
- 별칭 선택 입력
- 구매일 선택 입력
- 설치일 선택 입력
- 메모 선택 입력

### Step 4. 등록 완료

- 연결된 관리법 수준 안내
- 첫 관리법 보기
- 나중에 보기

완료 조건:

- 사용자는 정확한 모델을 검색해 3~4번의 주요 행동 안에 등록할 수 있다.
- 모델명을 모르는 사용자는 제품 종류만 선택해 등록할 수 있다.
- 뒤로 이동해도 이전 입력이 유지된다.
- 키보드가 열린 상태에서도 주요 버튼을 사용할 수 있다.

## 5.2 검색 품질 개선

지원할 검색:

- 정확한 모델명
- 모델명의 하이픈 및 공백 차이
- 영문 대소문자 차이
- 브랜드명
- 제품 종류
- 한국어 별칭
- 자주 쓰는 줄임말

예:

- `DCS-HM4AG-W`
- `DCS HM4AG W`
- `dcshm4agw`
- `음처기`
- `에코업 음식물`

검색 결과 정렬:

1. 모델명 정확히 일치
2. 모델명 부분 일치
3. 브랜드와 제품 종류 일치
4. 제품명 일치
5. 별칭 또는 키워드 일치
6. 일반 제품군

검색 결과에 표시할 정보:

- 제품 이름
- 브랜드
- 모델명
- 제품 종류
- 검수 상태
- 일치 수준
- 정보가 마지막으로 확인된 날짜

완료 조건:

- 정확한 모델명 검색 결과가 항상 첫 번째에 나온다.
- 공백과 하이픈을 무시한 검색이 동작한다.
- 검색 결과가 없는 경우 직접 등록과 정보 요청으로 연결된다.
- 검색 응답이 느릴 때 로딩 상태가 보인다.
- 서버 오류 시 로컬 폴백 또는 재시도 안내가 작동한다.

## 5.3 중복 제품 등록 방지

중복 판단 기준:

- 같은 공간 + 같은 카탈로그 제품 ID
- 같은 공간 + 같은 정규화 모델명
- 사용자가 지정한 동일 별칭

중복 발견 시 선택지:

- 기존 제품 보기
- 그래도 별도 제품으로 등록
- 공간만 변경

예외:

- 같은 모델의 에어컨이 방마다 있을 수 있다.
- 같은 모델의 공기청정기가 여러 대 있을 수 있다.

따라서 중복을 막기보다 경고하고 사용자가 선택하게 한다.

완료 조건:

- 실수로 같은 제품을 연속 등록하는 경우 안내가 나온다.
- 같은 모델을 여러 대 등록하는 것은 가능하다.
- 각 제품에 사용자가 구분할 별칭을 붙일 수 있다.

## 5.4 제품을 찾지 못한 경우

제공할 선택지:

- 제품 종류만 등록
- 브랜드와 모델명을 직접 입력
- 제품 정보 추가 요청
- 제품 라벨 사진 촬영 또는 선택

저장할 운영 데이터:

- 검색어
- 입력한 브랜드
- 입력한 모델명
- 제품 종류
- 검색 실패 시각
- 사용자가 일반 제품으로 등록했는지 여부

개인정보나 사진은 동의 없이 서버에 전송하지 않는다.

완료 조건:

- 검색 실패가 막다른 화면이 아니다.
- 사용자는 제품을 등록하거나 정보 요청을 보낼 수 있다.
- 운영자는 많이 요청된 미등록 제품을 확인할 수 있다.

# 6. Phase 2: 제품 카탈로그 품질 강화 (진행 중)

## 6.1 카탈로그 데이터 규칙 확정

필수 필드:

- 내부 제품 ID
- 제품 종류

- 핵심 스펙

선택 필드:

- 고객지원 URL
- AS 전화번호

- 대표 이미지
- 필터 및 소모품
- 설치 방식

금지:

- 출처 없는 숫자 스펙
- 공식 정보처럼 작성한 추정값
- 검수 상태 없는 사용자 제보
- 광고 여부가 표시되지 않은 제휴 상품

## 6.2 출처 모델 확장

제품 하나에 여러 출처를 연결할 수 있어야 한다.

출처 정보:

- 제목
- URL
- 출처 유형
- 게시 주체
- 확인일
- 어떤 정보의 근거인지
- 공식 여부
- 유효 여부

출처 유형:

- 공식 제품 페이지
- 공식 설명서
- 공식 영상
- 공식 고객지원
- 판매 페이지
- 가격비교 스펙
- 설치업체 자료
- 유사 제품 참고

완료 조건:

- 관리 단계와 스펙마다 어떤 출처를 참고했는지 추적할 수 있다.
- 깨진 링크와 오래된 출처를 재검토할 수 있다.

## 6.3 검수 절차

초안 작성:

- 조사자가 제품 기본 정보와 출처를 입력한다.
- 상태는 `draft`다.

1차 검수:

- 모델명과 제조사를 확인한다.
- 위험한 관리 단계가 없는지 확인한다.
- 상태를 `reviewed`로 변경한다.

2차 검수:

- 공식 자료와 주요 스펙을 교차 확인한다.
- 정확한 모델 전용 관리 정보인지 확인한다.
- 상태를 `verified`로 변경한다.

재검수:

- 마지막 확인일이 오래된 제품
- 링크가 깨진 제품
- 사용자 신고가 들어온 제품
- 제조사 안내가 변경된 제품

완료 조건:

- 검수 상태 없이 공개 API에 노출되는 제품이 없다.
- 누가 언제 무엇을 검수했는지 기록된다.

## 6.4 카탈로그 우선 확장

첫 확장 목표:

- 제품군 5개
- 제품군별 대표 브랜드 3개 이상
- 대표 모델 또는 제품군 가이드 5개 이상

우선 제품군:

1. 음식물처리기
2. 냉장고
3. 세탁기
4. 에어컨
5. 공기청정기

제품군별 조사 템플릿:

- 대표 관리 문제
- 평상시 관리
- 금지 행동
- 주요 소모품
- 공식 설명서 위치
- 모델 구분에 필요한 스펙
- 안전 또는 AS 전환 기준

완료 조건:

- 최소 25개의 검수된 카탈로그 항목이 있다.
- 제품군별 일반 관리법이 최소 하나 존재한다.
- 정확한 모델이 없을 때 일반 관리법으로 안정적으로 폴백된다.

권장 탭:

### 관리법

- 평상시 관리

- 사용자가 가장 자주 쓰는 행동이 상단에 있다.

## 7.2 관리법 표현 방식

각 관리법은 다음 구조를 가진다.

- 제목
- 적용 상황
- 예상 난이도
- 필요한 준비물
- 단계별 방법
- 금지 행동
- 중단 후 문의해야 하는 조건
- 근거 출처

관리법 유형:

- 일상 관리
- 정기 관리
- 문제 상황 관리
- 소모품 교체
- 전문가 문의

완료 조건:

- 한 관리법 안에서 준비물, 단계, 주의사항이 섞이지 않는다.
- 위험하거나 제품 분해가 필요한 단계는 사용자 관리법으로 제공하지 않는다.

## 7.3 출처와 신뢰도 UX

사용자에게 보여줄 표현:

- `모델 공식 정보`
- `제조사 공식 영상`
- `동일 브랜드 유사 모델 참고`
- `같은 방식의 제품 참고`
- `일반 제품군 관리법`
- `사용자 입력 정보`

내부 검수 상태인 `draft`, `reviewed`, `verified`를 그대로 노출할 필요는 없다.

사용자에게 의미 있는 표현으로 변환한다.

완료 조건:

- 사용자는 이 관리법이 자기 모델 전용인지 즉시 알 수 있다.
- 출처를 직접 열 수 있다.
- 출처가 없는 일반 팁은 일반 관리법으로 표시된다.

# 13. Phase 9: 선택적 알림

- 아직 안 했어요
- 실패했어요

권장 표현:

- 확인해볼 시기예요

- 미완료 일정으로 계속 누적하지 않는다.
- 홈에서 부드러운 추천으로만 표시한다.
- 사용자가 무시하거나 다음으로 넘길 수 있다.

완료 조건:

- 알림을 끈 제품에는 알림이 오지 않는다.
- 알림을 눌렀을 때 정확한 제품 관리법으로 이동한다.
- 놓친 알림이 죄책감 유발 목록으로 쌓이지 않는다.

# 15. Phase 11: 커뮤니티와 문제 진단

이 단계는 제품 검색, 상세, 기록이 충분히 완성된 후 시작한다.

## 15.1 제품별 사용자 팁

- 특정 모델에 연결
- 공식 정보와 분리
- 신고 기능
- 위험한 팁 차단
- 유용함 평가

## 15.2 문제 진단

구조:

1. 증상 선택
2. 안전 관련 질문
3. 사용자가 할 수 있는 범위 판단
4. 관리법 또는 AS 연결

결과:

- 직접 관리 가능
- 설명서 확인
- 소모품 교체
- 사용 중단
- 전문가 문의

완료 조건:

- 안전 관련 증상은 가장 먼저 확인한다.
- 앱이 고장 진단을 확정적으로 단정하지 않는다.
- 출처 없는 해결책을 자동 생성하지 않는다.

# 18. 구체적인 다음 4개 스프린트

## Sprint A. 제품 등록 흐름 재설계

목표:

> 제품 검색과 등록을 단계형으로 바꾸고 중복과 검색 실패를 처리한다.

작업:

- 제품 등록 화면을 별도 전체 화면으로 분리
- 검색과 직접 등록 경로 분리
- 정확한 검색 결과 카드 설계
- 모델 일치 수준 표시
- 중복 제품 경고
- 제품 별칭
- 검색 실패 시 정보 요청 진입
- 키보드와 작은 화면 검증

완료 기준:

- 정확한 모델 등록
- 모델 없는 일반 등록
- 중복 제품 등록
- 검색 실패 후 직접 등록

네 가지 흐름이 모두 테스트된다.

## Sprint B. 제품 상세 정보 구조 재설계

목표:

> 긴 단일 목록을 목적별 탭 구조로 바꾼다.

작업:

- 관리법 탭
- 소모품 탭
- 기록 탭
- 제품 정보 탭
- 출처 수준 배지
- 설명서와 영상 링크
- 관리 기록 빠른 추가

완료 기준:

- 주요 정보가 2번 이하의 탭 또는 스크롤 행동으로 접근 가능하다.
- 작은 화면과 큰 글자에서 겹침이 없다.

## Sprint C. 제품별 관리 기록

목표:

> 청소 완료 한 줄 기록을 실제 관리 이력으로 바꾼다.

작업:

- 기록 모델 분리
- 기록 유형
- 메모
- 사용 용품
- 문제 증상과 결과
- 제품 상세 이력
- 전체 기록 필터
- 수정과 삭제

완료 기준:

- 제품 하나를 관리하고 기록한 뒤 앱을 재실행해도 이력이 유지된다.
- 기록을 수정하고 삭제할 수 있다.

## Sprint D. 카탈로그 운영 기반

목표:

> 제품 데이터를 지속적으로 추가할 수 있는 기준과 도구를 만든다.

작업:

- 출처 다중 연결
- 제품군 조사 템플릿
- 검수 체크리스트
- 검색 실패 로그 API
- 제품 추가 요청 API
- 첫 5개 제품군 데이터 확장
- 서버와 앱 API 계약 테스트

완료 기준:

- 최소 25개 검수 항목
- 제품군별 일반 관리법
- 검색 실패 수집
- 검수되지 않은 데이터 미노출

# 20. 작업 완료 정의

기능은 코드가 작성됐다고 완료된 것이 아니다.

다음 조건을 만족해야 완료로 본다.

- 요구한 사용자 흐름이 실제로 동작한다.
- 로딩, 빈 상태, 오류 상태가 있다.
- 저장 후 앱을 재실행해도 데이터가 유지된다.
- 작은 화면에서 UI가 잘리지 않는다.
- 키보드 입력이 정상 동작한다.
- 정적 분석이 통과한다.
- 관련 자동 테스트가 추가되고 통과한다.
- 에뮬레이터 또는 실제 기기에서 확인한다.
- 사용자에게 보여주는 문구가 제품 관리 방향과 맞는다.
- 출처가 필요한 정보에는 출처가 있다.

# 21. 새로운 아이디어가 생겼을 때

새 기능을 즉시 개발하지 않는다.

먼저 아래 항목을 짧게 작성한다.

- 해결하려는 사용자 문제
- 주 사용 상황
- 핵심 사용자 흐름과의 연결
- 필요한 제품 데이터
- 안전 또는 신뢰도 위험
- 지금 개발해야 하는 이유
- 기존 기능으로 해결할 수 없는 이유
- 완료 여부를 판단할 지표

다음 중 하나라도 명확하지 않다면 아이디어 목록에 보관하고 핵심 개발을 계속한다.
